Django安全开发指南[^1]

本指南适用于Linux系统。同样的技术设置也可用于Windows系统。

这些说明适用于在生产模式下运行的应用程序。开发/测试实例不需要加固，因为这会增加测试难度。

配置加固

本章节对Django的配置文件进行加固，文件名为：global_settings.py。在这个配置文件中可以修改Django的默认配置。该文件在Django安装路径的固定目录下，例如： \Django-1.10.2\django\conf

当用户创建一个新的Django项目时，会生成一个新的settings.py，可以在该文件中修改新项目的配置。

注意，有些说明并非适用于所有情况。请阅读每一条建议，确定是否有需要注意的限制或事项。

  • 禁用调试

    重要性：强制

    在生产环境中启用调试会泄露相关应用程序的信息，包括源代码，设置和局部变量。调试功能只能在开发模式下启用。

    配置要求

    在应用程序配置中将DEBUG设置为False。

    影响：无

  • 将密钥设置为随机值

    重要性：强制

    密值用于诸如会话标识符和密码重置令牌的加密操作。密值必须是随机的、唯一的，避免发生由于使用默认值而损害操作的情况。

    配置要求

    在应用程序配置中将SECRET_KEY设置为随机值。涉及三层秘钥管理结构。

    影响：无

  • 定义使用的主机名/域名

    重要性：可选

    为了防止HTTP主机头攻击，通过配置Django，只接受请求中允许的主机名/域名。

    配置建议

    在应用程序配置中，将ALLOWED_HOSTS设置为Django服务的主机名/域名。具体值取决于应用程序和它应该操作的域。

    如果开发团队预先不知道域，那么该配置应该当作一项部署任务进行备案，以便在实时环境中部署该应用程序之前进行更改。如果开发的Django服务仅在内网中使用，接受的请求没有域名或不适合配置域名的情况，可以不配置该项。

    影响：无

  • 跨站请求伪造(CSRF)和会话Cookie使用“安全”标志

    重要性：可选

    让Django自动设置Cookie上的“安全”标志是可行的，而且强烈建议这样做。在这种场景下，不使用HTTPS的连接不会发送Cookie。这能防止会话劫持，以及其他因无意使用HTTP发送敏感Cookie而造成的攻击。

    不是明确需要通过HTTP发送的Cookie必须使用“安全”标志。然而，这同样可以由系统的不同部分（例如，反向代理或其他系统）来实现。因此，当且仅当某些其他系统已经使用“安全”标志或该标志总是由应用程序直接添加时，这项要求才是可选的。

    CSRF Cookie用于保护使用Django助手功能创建的表单。如果没有使用其他CSRF机制，这项要求是强制的，但是如果一些其他系统使用CSRF保护，则可以自愿选择。

    配置建议

    在应用程序配置中将CSRF_COOKIE_SECURE和SESSION_COOKIE_SECURE设置为True。

    影响：无

  • 使用HTTP严格传输安全（HSTS）

    HSTS的作用是强制客户端（如浏览器）使用HTTPS与服务器创建连接。服务器开启HSTS的方法是，当客户端通过HTTPS发出请求时，在服务器返回的超文本传输协议（HTTP）响应头中包含Strict-Transport-Security字段。非加密传输时设置的HSTS字段无效。比如，https://example.com/ 的响应头含有Strict-Transport-Security: max-age=31536000; includeSubDomains。这意味着两点:

    1. 在接下来的31536000秒（即一年）中，浏览器向example.com或其子域名发送HTTP请求时，必须采用HTTPS来发起连接。比如，用户点击超链接或在地址栏输入 http://www.example.com/ ，浏览器应当自动将 http 转写成 https，然后直接向 https://www.example.com/ 发送请求。
    2. 在接下来的一年中，如果 example.com 服务器发送的TLS证书无效，用户不能忽略浏览器警告继续访问网站。

    更多参考资，点击链接

    该功能对智库的影响：

    • 前后台很多页面是要求HTTP才能访问、下载的，这些页面需要通查通该
    • 测试环境服务器、测试文件服务器、正式文件服务器全都没有配置HPPTS，后续全都要配置上才可以。

重要性：可选

HTTP严格传输安全是一个HTTP头，用于通知客户端浏览器只连接到使用HTTPS的应用程序。如果应用程序没有HTTP部分，应始终使用该HTTP头。该要求可选，因为它并不适用于某一部分仅为HTTP服务器的系统。

配置建议

在应用程序配置中将SECURE_HSTS_SECONDS至少设置为31536000（一年）。包括子域的情况下，请将SECURE_HSTS_INCLUDE_SUBDOMAINS设置为True。

注：这两个配置项在Django 1.8或以后版本中支持，如果产品使用Django 1.8以前版本则不涉及这个配置。

影响：无

运行时配置加固

  • 使用Python散列（hash）随机值

    python的字符串hash算法并不是直接遍历字符串每个字符去计算hash，而是会有一个secret prefix和一个secret suffix，可以认为相当于是给字符串加盐后做hash，可以规避一些规律输入的情况

    显然这个secret前后缀的值会直接影响计算结果，而且它有一个启动时随机生成的机制，只不过，在2.x版本中，这个机制默认是关闭的，前后缀每次启动都设置为0，除非你改了相关环境变量来要求随机，而在3.x中修改了默认行为，如果你不配置环境变量，则默认是随机一个前后缀值，这样每次启动都会不同

    这个环境变量是PYTHONHASHSEED，无论在2.x还是3.x中，配置为一个字符串的正整数，将作为随机种子；配置为‘0’，则secret前后缀默认清零（和2.x默认行为就一样了），配置为空串或“random”，则表示让进程随机生成（和3.x默认行为一样）

    官方文档的解释：If this variable is not set or set to random, a random value is used to seed the hashes of str, bytes and datetime objects.

    If PYTHONHASHSEED is set to an integer value, it is used as a fixed seed for generating the hash() of the types covered by the hash randomization.

    Its purpose is to allow repeatable hashing, such as for selftests for the interpreter itself, or to allow a cluster of python processes to share hash values.

    The integer must be a decimal number in the range [0,4294967295]. Specifying the value 0 will disable hash randomization.

    重要性：可选

    Python hash随机化能防止某些类型的拒绝服务（DoS）攻击，DoS攻击利用dict类型中最坏用例构造。这尤其会影响Django应用程序，应加以防范。

    配置建议

    调用使用-R标志运行Django的Python进程或将PYTHONHASHSEED的环境变量设置为random。能否实现取决于环境设置，且应该充分验证。

    影响：无

安全开发指导

  • 保护Django 管理员（admin）

    在生产环境中从INSTALLED_APPS设置中删除django.contrib.admin，以禁用Django admin，并从urls.py中删除the admin.site.urls条目。

    如果您想使用Django admin，请将其托管在其他域，并确保其只能通过内部网络或VPN等进行访问。

  • 保护Django设置文件

    创建新的Django项目时，会生成一个带有新SECRET_KEY的settings.py。此密钥用于Django中的加密操作，该文件还包含其他敏感信息，例如IP地址和外部系统的认证凭据。settings.py决不能泄露，应设置合理的访问权限（建议设置权限为700）。

    -rwx—— ( 700) 只有拥有者有读、写、执行权限。

    一共有十位数，其中：最前面那个 - 代表的是类型

    中间那三个 rw- 代表的是所有者（user）拥有的权限

    然后那三个 r– 代表的是组群（group）拥有的权限

    最后那三个 r– 代表的是其他人（other）拥有的权限

    然后

    r 表示文件可以被读（read）

    w 表示文件可以被写（write）

    x 表示文件可以被执行（如果它是程序的话）

    - 表示相应的权限还没有被授予

    之后的root root 就是用户拥有者了

    需要注意的是：查看文件权限的目录问题: 如果有文件夹 /a/b/c

    那么执行 ls -l /a/b 查看权限的文件并不是b，而是查看的c的权限。

  • 使用Django内置认证，授权和会话管理

    Django自带内置认证，授权和会话管理系统，可放心使用。详细内容请参考：https://docs.djangoproject.com/es/1.10/topics/auth/。

    使用Django内置认证，授权和会话管理有以下好处：

    • 用户口令使用PBKDF2进行安全哈希处理。更多关于Django口令管理信息，请参考：https://docs.djangoproject.com/es/1.10/topics/auth/passwords/。

    • 会话标识符强且随机。

    • 会话有效期默认为两周，可以通过SESSION_COOKIE_AGE设置修改这一期限：https://docs.djangoproject.com/en/1.10/ref/settings/#std:setting-SESSION_COOKIE_AGE。

    • 免受会话固定攻击。更多关于会话固定攻击的信息，请参考：https://www.owasp.org/index.php/Session_fixation。

    • 使用内置Django模型来管理用户、群组和权限。

    • 使用内置Django装饰器对视图进行访问控制。

    • 使用内置Django admin管理用户、群组和权限。

    • 使用内置Django表单管理注册、认证和口令。

    • 最重要的是，您不必再施行自己的会话管理、认证或授权机制，这些机制可能有风险。

  • 对外部输入进行校验

    永远不要默认信任用户输入。验证来自不可信源的所有输入（例如请求参数、Cookie和信息头）。推荐使用白名单校验。

    在Django中，可用以下方法验证输入。

    • 对于URL参数（有时称为GET参数），第一级验证是在urls.py中设置限制性正则表达式。除此之外，还需在视图中输入验证逻辑。因为：

    • urls.py文件只允许指定一个正则表达式，但有时需要进行更复杂的验证。

    • 视图有时可以从（例如）另一个视图中进行调用，不需通过urls.py，这会有效绕过URL正则表达式验证。

    • 对于body参数（有时称为POST参数），最简单的方法是使用Django的表单（forms）和验证器（validators）。关于表单和验证器，请分别参考：https://docs.djangoproject.com/en/1.10/topics/forms/和https://docs.djangoproject.com/en/1.10/ref/forms/validation/. Django提供了一套很好的内置验证器，您也可以编写自己的验证器。

    • 对于已处理的Cookie和其他HTTP头，请在视图中输入验证逻辑。

      此外，可以添加客户端验证以提升用户体验，如果您记录了潜在恶意请求的安全日志，请缩小误报率。

  • 使用Django模板，防止XSS

    XSS是一种网络攻击，受控数据在网页上显示为超文本标记语言（HTML），而不是文本。这种攻击使攻击者能在受害用户的Web浏览器中执行恶意的Javascript代码。关于跨站点脚本攻击，请参考：https://www.owasp.org/index.php/Cross-site_Scripting_%28XSS%29。

    默认情况下，Django的模板系统对**>，<，&，‘和’**字符进行HTML编码。只要遵循一些基本规则，这种自动编码可以有效防止XSS。

    • 使用模板系统。当编写视图时，使用django.shortcuts.render从模板呈现HTML。不要使用其他方法，例如，用django.http.HttpResponse将HTML返回到客户端。Django只自动保护从模板呈现的HTML。

    • 不要通过{% autoescape off %}，|safe或者在上下文（context）中设置autoescape=False关闭自动编码。

    • 仅在安全上下文中呈现模板变量（例如****）：

      • 内部HTML标记，除了脚本标记或可用于执行代码的其他HTML标记。例如：<p>搜索：****</p>

      • 内部有引号的HTML属性。例如：<input type=”text” value=”“/>

      • 内部有引号的Javascript字符串。例如：<script>var userName = ““;</script>

    • 永远不要在不安全的上下文中呈现模板变量，例如：

      • 内部DOM事件处理程序（甚至没有引号）。例如：<p onclick=”alert(‘‘)”>Click to show username!</p>

      • 内部客户端代码，没有引号（例如：Javascript/VBScript）。例如：<script>var userName = ;</script>

      • 内部没有引号的HTML属性。例如：<input type=”text” value=/>

      • 内部不安全Javascript函数，例如：eval，setTimeout或setInterval。

    • 内部HTML注释。例如：<!– Username: –>

    • 内部HTML属性名称。例如：<a =”test”>test

    • 内部HTML标记名称。例如：<>

    • URL编码URL地址中的模板变量。例如：<a href=”/users/“>Show user</a>

    • 尽可能严格地实施CSP（内容安全策略）头，使应用程序仍然运行。如果可以，从头开始编写应用程序，以便实施CSP时可以禁用除了从安全域加载的Javascript之外的所有客户端代码。更多关于CSP的信息，请参考：https://www.owasp.org/index.php/Content_Security_Policy。要在Django中实现CSP，可以使用Mozilla的**django-csp**软件包：https://github.com/mozilla/django-csp。

    • 实施X-XSS-Protection头，确保Web浏览器的内置XSS保护功能未关闭。要实施X-XSS-Protection头，在 settings.py里设置SECURE_BROWSER_XSS_FILTER = True。

    • 使用HttpOnly标志保护会话Cookie。当使用Django的会话管理时，可以通过在settings.py中设置SESSION_COOKIE_HTTPONLY = True来实现。

    有关防止XSS漏洞的更多信息，请参考OWASP速查表：https://www.owasp.org/index.php/XSS_%28Cross_Site_Scripting%29_Prevention_Cheat_Sheet。

  • 在POST请求中验证CSRF令牌，防止CSRF

    CSRF是一种攻击，发动该攻击时，恶意或受损网站向受害用户会话中的目标应用程序发出请求。关于CSRF攻击，请参考：https://www.owasp.org/index.php/Cross-Site_Request_Forgery_%28CSRF%29。

    • 不要使用POST之外的其他HTTP方法进行更改。Django CSRF令牌只保护POST方法，GET和HEAD等其他方法不受保护。因此，做更改时只能使用POST方法，其他方法应该是“只读”。

    • 不要通过，如csrf_exempt装饰器禁用CSRF保护。

  • 使用Django提供的ORM，防止SQL注入

    SQL注入是一种攻击，将用户输入合并到SQL查询中，使攻击者有效地控制查询结构。攻击者可以利用SQL注入漏洞读取和写入数据库，读取和写入系统文件，有时甚至执行操作系统命令。更多关于SOL注入攻击，请参考：https://docs.djangoproject.com/es/1.10/topics/db/。

    Django的ORM（对象关系映射器）能防止SQL注入，但需要遵循以下基本规则：

    • 使用Django提供的ORM。不要使用其他任何方法访问数据库。

    • 不要在Django ORM中使用不安全的方法做原始查询。不安全的方法指 **raw()和extra()**。

      遵循以上原则，就可避免SQL注入。更多关于Django ORM的信息，请参考：https://docs.djangoproject.com/es/1.10/topics/db/。

  • 使用django.core.mail，防止电子邮件头注入

    电子邮件头注入是一种攻击，攻击者通过在参数中添加换行符，将自己的信息头注入电子邮件头。更多关于电子邮件头注入攻击的信息，请参考：https://en.wikipedia.org/wiki/Email_injection。

    为了防止电子邮件头注入攻击，使用django.core.mail中的Django邮件函数发送电子邮件。

  • 防止中间人（MITM）攻击

    在MITM攻击中，攻击者驻留在受害者的Web浏览器和Web服务器之间。如果网站流量未适当加密，攻击者可以读取、拦截甚至对其进行修改。更多关于中间人攻击的信息，请参考：https://www.owasp.org/index.php/Man-in-the-middle_attack。

    为保护网站流量不被攻击者拦截，可以：

    • 仅通过加密的HTTPS提供内容。将明文HTTP请求重定向到一个https:// URL。在Django中，这可通过在settings.py中设置SECURE_SSL_REDIRECT = True来实现。

    • 使用安全（ secure）标志保护会话和CSRF Cookie。该标志告知网络浏览器，Cookies只能通过加密的HTTPS发送，绝不能通过明文HTTP发送。在Django中，这可通过在settings.py中添加SESSION_COOKIE_SECURE = True和CSRF_COOKIE_SECURE = True来实现。

    • 使用Strict-Transport-Security头，防止Web浏览器（在第一次看到Strict-Transport-Security头之后）通过未加密的连接访问网站。在使用Strict-Transport-Security头之前，请确保您了解其工作原理，以防阻止用户访问您的网站。更多关于Strict-Transport-Security的信息，请参考：https://www.owasp.org/index.php/HTTP_Strict_Transport_Security_Cheat_Sheet。

      一旦了解了其工作原理，您可以通过在settings.py中设置以下选项，在网站的域和子域将HSTS设置为一年：

    • HSTS有一个缺陷。在浏览器看到Strict-Transport-Security头之前，它允许Web浏览器通过未加密的连接访问页面一次。要解决该缺陷，请将您的网站提交到HSTS预加载列表：https://hstspreload.appspot.com/。

    • 确保Web服务器仅支持安全的SSL / TLS协议和密码。更多详细信息，请参阅Web服务器的加固指南。
